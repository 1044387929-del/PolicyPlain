# 项目实施流程与顺序（参考文档）

本文档说明本笔试项目从立项到交付的**推荐推进顺序**，可与《项目需求分析》配合使用。小项目不必每一步都产出厚文档，可按「最小够用」执行。

---

## 1. 总览：先想清楚，再定形状，再写代码

```text
需求分析 → 概要设计（接口+页面）→ 开发 → 自测联调 → 演示与交付整理
```

**原则**：先对齐「验收时要点哪几下、看到什么」，再冻结接口与数据结构，最后填充实现。避免前后端各写各的、返工。

---

## 2. 各阶段说明

### 阶段 ① 需求分析


| 项         | 说明                                |
| --------- | --------------------------------- |
| **目的**    | 明确为谁解决什么问题、做哪些功能、**明确不做**什么、如何验收。 |
| **产出物**   | 《项目需求分析》（或等价一页纸）。                 |
| **是否可省**  | 不建议省略；可压缩篇幅，但验收标准建议保留。            |
| **本项目对应** | `docs/项目需求分析.md`                  |


---

### 阶段 ② 概要设计


| 项         | 说明                                                                       |
| --------- | ------------------------------------------------------------------------ |
| **目的**    | 定系统怎么拆：页面有哪些区块、后端有哪些接口、AI 在哪一步介入、数据怎么进出。                                 |
| **产出物**   | 简图或清单即可：路由表、请求/响应 JSON 字段、主要页面线框（手绘或文字分块均可）。                             |
| **是否可省**  | 不能完全省；至少要有**接口约定**（可与阶段 ③ 合并）。                                           |
| **本项目对应** | 前端：粘贴区、主题标签（若有）、结果卡片分区、加载/失败/免责展示；后端：`POST /api/...` 类解读接口、校验、超时、调用 LLM。 |


---

### 阶段 ③ 详细设计


| 项        | 说明                                         |
| -------- | ------------------------------------------ |
| **目的**   | 细化错误码、边界情况、Prompt 输出 JSON Schema、日志与脱敏规则等。 |
| **产出物**  | 接口注释、OpenAPI、或设计笔记中的「边界与错误」小节。             |
| **是否可省** | 时间紧时可与开发同步写；**结构化 AI 输出**建议尽早写死字段，减少返工。    |


---

### 阶段 ④ 开发


| 项        | 说明                                                                                 |
| -------- | ---------------------------------------------------------------------------------- |
| **目的**   | 实现可演示的完整链路。                                                                        |
| **推荐顺序** | 1）**数据库建表**（PostgreSQL + **Alembic 迁移**）→ 2）**注册/登录与 JWT** → 3）受保护的政策解读接口（可先 mock）→ 4）前端登录态与路由守卫 → 5）接入真实 LLM（可选）→ 6）历史列表/详情 → 7）适老化与免责、错误提示。详见 `docs/技术实施步骤.md`。 |
| **技术栈**  | FastAPI（后端）、Vue（前端）；AI 层（如 LangChain）在服务端调用，密钥勿进前端。                                |


---

### 阶段 ⑤ 自测与联调


| 项       | 说明                                                |
| ------- | ------------------------------------------------- |
| **目的**  | 对照需求中的验收项，用真实或拟真政策文本反复试。                          |
| **关注点** | 空输入、超长文本、超时、模型胡编数字/日期时是否被约束或标注「原文未提及」；免责声明是否始终可见。 |


---

### 阶段 ⑥ 部署与演示环境（按笔试要求）


| 项        | 说明                                            |
| -------- | --------------------------------------------- |
| **目的**   | 验收当天可稳定打开或一键启动。                               |
| **产出物**  | `README.md`：环境版本、依赖安装、前后端启动命令、环境变量说明（不写真实密钥）。 |
| **是否可省** | 若允许仅本地演示，至少保证 README 可复现。                     |


---

### 阶段 ⑦ 交付物整理


| 项        | 说明                                                 |
| -------- | -------------------------------------------------- |
| **目的**   | 让阅卷人一眼看到「需求—设计—实现—如何运行」。                           |
| **常见组合** | 源码 + README + 需求/流程文档（本文档与需求分析）+（可选）演示录屏或样例输入输出截图。 |


---

## 3. 与「正式大项目」的对比（心理预期）


| 正式项目常见      | 本笔试建议                |
| ----------- | -------------------- |
| 多轮评审、冗长 BRD | 需求分析 1 份，控制在可读篇幅     |
| 独立详设评审      | 接口 + 页面结构合并为 1 份笔记即可 |
| 测试用例库       | 2～3 条手工验收路径 + 边界试几次  |
| 长期运维方案      | 可写「后续若上线需…」一句带过      |
