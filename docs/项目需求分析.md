## 2. 项目背景与问题陈述

### 2.1 背景

各级人社、医保部门发布的政策、办事指南、通知公告往往篇幅长、术语多、交叉引用多。部分老年人阅读能力、信息检索能力下降，容易遗漏办理时限、材料清单、线上线下渠道等关键信息，家属协助成本高。

### 2.2 待解决问题（核心价值）

在**不替代官方解释与人工服务**的前提下，为老年人及其家属提供一种**辅助阅读**工具：将用户提供的政策类文本，转化为**结构清晰、用语通俗、步骤明确**的摘要与行动指引，降低误读与漏办风险。

### 2.3 项目目标（可验收）

1. **已登录用户**可输入一段与**社保 / 医保 / 养老保险**相关的政策或通知正文（粘贴为主）；未登录用户须先完成注册/登录（见 F0）。
2. 系统输出**结构化解读结果**（要点、条件、材料、渠道、时限等），用语适合老年人阅读（短句、少术语或术语有注释）。
3. 明确标注**信息来源与不确定性**，引导用户在重大事项上咨询官方窗口或热线。
4. 用户解读记录**持久化保存**（关联登录账号），支持按时间查看历史记录列表与单条详情（见 F5）。

## 3. 用户角色与使用场景

### 3.1 用户角色


| 角色            | 诉求                     | 备注                   |
| ------------- | ---------------------- | -------------------- |
| 老年人本人         | 快速抓住「与我有关吗、要做什么、什么时候做」 | 主用户；界面需大字号、高对比、少操作步骤 |
| 家属协助者         | 代为粘贴文本、核对解读是否遗漏要点      | 次要用户；可与老年人共用同一终端     |
| 已登录用户（通用）   | 安全使用个人历史记录；跨多次访问查看过往解读 | 须完成注册/登录；密码等凭据由系统安全存储   |
| 运营/演示人员（笔试场景） | 准备样例、展示链路              | 非长期角色                |


### 3.2 典型场景

1. **新用户首次使用**：注册账号并登录后，进入政策解读主界面。
2. **社区/单位转发的通知**：用户粘贴全文，需要提取办理事项与截止时间。
3. **办事指南片段**：用户粘贴某业务说明，需要材料清单与线上/线下办理路径。
4. **政策条文节选**：用户粘贴若干条，需要「适用对象 + 影响 + 下一步」式归纳。
5. **回看历史**：用户从「我的解读记录」中打开某一条，查看当时保存的结构化结果（及按需展示的原文）。

### 3.3 非目标场景（本期不做）

- 自动爬取政府网站全站政策库、持续同步更新。
- 代替官方作出资格认定、待遇核算、参保状态查询（需对接政务系统与实名鉴权）。
- 医疗诊断、用药建议等与政策解读无关的医疗服务。

## 4. 功能需求

### 4.1 功能总览

```text
[F0] 用户注册、登录、登出与会话鉴权
[F1] 政策文本输入与校验
[F2] 政策白话解读（核心）
[F3] 结果展示与适老化阅读
[F4] 安全提示与免责声明
[F5] 解读历史持久化与查询（按登录用户隔离）
```

### 4.2 F0：用户注册、登录、登出与会话鉴权

| 需求 ID | 描述 | 优先级 |
| --- | --- | --- |
| F0.1 | 用户可通过**邮箱** + **密码**注册账号 | P0 |
| F0.2 | 用户可登录；登录成功后获得**有期限的会话令牌**（如 JWT）或等价机制，用于后续 API 鉴权 | P0 |
| F0.3 | 用户可登出；客户端丢弃令牌；服务端可按实现使令牌失效（黑名单可选，笔试可简化） | P1 |
| F0.4 | **政策解读**、**历史记录查询**等核心业务接口须**鉴权**：未携带有效令牌时拒绝访问 | P0 |
| F0.5 | 密码须**哈希存储**（禁止明文）；传输在部署环境使用 **HTTPS**（本地开发可例外说明） | P0 |
| F0.6 | 注册时对**邮箱**唯一性校验；重复注册返回明确错误提示 | P0 |

### 4.3 F1：政策文本输入与校验


| 需求 ID | 描述                                             | 优先级 |
| ----- | ---------------------------------------------- | --- |
| F1.1  | 支持多行文本粘贴，最小/最大长度限制（防止空输入与过长请求）                 | P0  |
| F1.2  | 可选：让用户选择大致主题标签（社保综合 / 医保 / 养老保险），用于引导模型侧重与前端展示 | P1  |
| F1.3  | 基础校验：非法字符、明显非政策类内容可提示「建议粘贴政策原文」，但不强制阻断（避免误杀）   | P1  |


### 4.4 F2：政策白话解读（核心）


| 需求 ID | 描述                                                                        | 优先级 |
| ----- | ------------------------------------------------------------------------- | --- |
| F2.1  | 输出须为**结构化字段**（见 4.4.1），便于前端卡片化展示，而非单一大段散文                                 | P0  |
| F2.2  | 对原文中**未写明**的推断内容须单独标注为「推测」或「原文未提及」，不得伪装为确定事实                              | P0  |
| F2.3  | 对涉及**金额、比例、日期、地区差异**的内容：若原文有则逐字忠实引用关键数字与日期；若无则不得编造                        | P0  |
| F2.4  | 输出中应包含**「建议您进一步核实」**类引导，指向官方渠道类型（热线、窗口、政务 App/小程序等——仅作类型提示，不伪造具体号码除非来自原文） | P0  |
| F2.5  | 语言风格：短句优先；关键术语首次出现可附一句白话注释                                                | P1  |


#### 4.4.1 建议的结构化输出字段（需求级，非接口定稿）

- **一句话摘要**：谁、什么事、何时前要注意。
- **与我相关的条件**：分条列出适用/不适用情形（原文依据简要指向）。
- **需要准备的材料**：清单式；原文未列则标明「原文未列材料清单」。
- **办理方式与渠道**：线上/线下/委托等；以原文为准。
- **重要时间节点**：报名、申报、缴费、待遇领取等；无则注明无。
- **常见误读提醒**：基于原文易混淆点列 1～3 条（无则省略）。
- **原文未覆盖事项**：列出用户可能关心但原文没写的问题，建议官方咨询。

### 4.5 F3：结果展示与适老化阅读


| 需求 ID | 描述                                              | 优先级 |
| ----- | ----------------------------------------------- | --- |
| F3.1  | 结果分区展示（摘要 / 材料 / 时间 / 渠道等模块），支持折叠次要信息           | P0  |
| F3.2  | 适老化：默认较大字号、足够行距与对比度；登录后主路径：粘贴 → 解读 →（可选）查看历史 | P0  |
| F3.3  | 加载中与失败状态有明确提示；失败时可重试                            | P0  |
| F3.4  | （可选）支持打印友好样式或「复制全部要点」                           | P2  |


### 4.6 F4：安全提示与免责声明


| 需求 ID | 描述                                         | 优先级 |
| ----- | ------------------------------------------ | --- |
| F4.1  | 全站固定展示：本工具为**阅读辅助**，不构成法律、行政或财务意见          | P0  |
| F4.2  | 涉及待遇享受、转移接续、补缴等敏感事项，突出提示**以当地最新政策及经办答复为准** | P0  |
| F4.3  | 不在界面中伪造红头文件、公章、政府域名链接                      | P0  |


### 4.7 F5：解读历史持久化与查询

| 需求 ID | 描述 | 优先级 |
| --- | --- | --- |
| F5.1 | 每次成功的政策解读在服务端**写入数据库**，至少包含：`user_id`、请求时间、`topic`、**结构化解读结果**（JSON）；**原文全文**是否一并持久化由实现选定，须在用户可见的隐私说明中披露 | P0 |
| F5.2 | 用户可查看**本人**历史记录列表（按时间倒序，支持分页或「加载更多」，实现阶段定） | P0 |
| F5.3 | 用户可打开单条历史记录，查看已保存的结构化结果；若存储了原文，则一并展示 | P0 |
| F5.4 | 用户**不可**查看其他用户的历史记录（数据隔离） | P0 |

## 5. 非功能需求

### 5.1 性能

- 单次解读在常规模型与网络条件下，**目标交互可接受时间**为数十秒内；超时需友好失败提示。
- 对输入长度设上限，避免阻塞与过高成本。

### 5.2 可用性与无障碍

- 键盘可操作主要流程；焦点状态可见。
- 色彩不作为唯一信息区分手段（配合图标或文案）。

### 5.3 安全与隐私

- **账号与凭据**：密码仅保存哈希；登录令牌有期限；禁止在日志中输出明文密码或完整令牌。
- **解读内容落库**：用户粘贴的政策文本可能含姓名、身份证号、手机号等敏感信息。持久化前须在界面提供**隐私提示**；实现上须限制访问范围为**记录所属用户本人**（见 F5.4），并对管理/运维日志做**脱敏**（避免打印全文）。
- **数据删除（可选 P2）**：实现阶段可决定是否提供「删除单条历史」或「清空我的历史」；若不做，须在文档中说明。
- 传输使用 HTTPS（部署环境要求，本地开发可注明例外）。

### 5.4 可维护与扩展

- 前后端分离，解读逻辑与展示解耦，便于替换模型与调整 Prompt/策略。
- 配置项（模型名、超时、长度上限、**数据库连接**、**JWT 密钥与过期时间**）外置，不写死在代码中（需求级约定）。

## 6. 技术架构约束


| 层次   | 技术           | 说明                                      |
| ---- | ------------ | --------------------------------------- |
| 前端   | Vue          | 单页应用；登录态、路由守卫、输入展示、适老化样式与错误处理           |
| 后端   | FastAPI      | 认证与鉴权、解读 API、历史 API；参数校验、调用 AI、统一错误码与超时 |
| 持久化  | 关系型数据库      | 存储用户与解读历史；当前实现默认 **PostgreSQL**（`asyncpg`），见 `policy-backend/settings` |
| AI   | 待定实现         | 需求上要求：结构化输出、拒答/标注规则、合规话术                |

## 7. 合规与内容风险

1. **忠实性**：解读必须紧扣用户提供的文本，禁止编造地方政策版本、禁止虚构办理渠道与联系方式。
2. **地域与时效**：政策具地方性与时效性；产品须在界面与结果中反复强调**以当地最新规定为准**。
3. **责任边界**：产品定位为辅助工具，不承诺解读完整性与结论正确性。

## 8. 验收标准


| 编号  | 验收项                                     |
| --- | --------------------------------------- |
| AC1 | 使用提供的 2～3 段真实或拟真政策样例，能稳定返回结构化结果且主要字段非空  |
| AC2 | 样例中含模糊或缺失信息时，系统能标明「原文未提及」或「推测」，无凭空数字与日期 |
| AC3 | 前端完成适老化基本布局（大字、清晰分区、主要流程可演示）            |
| AC4 | 免责声明在首屏或结果页显著位置可见                       |
| AC5 | 技术栈符合约定：FastAPI + Vue + 关系型数据库 联调通过   |
| AC6 | 可完成注册、登录；未登录无法调用解读与历史接口；密码不以明文存储与传输（部署环境 HTTPS） |
| AC7 | 登录后产生的解读可在历史列表中查看，且仅能查看本人数据           |

## 9. 假设与依赖

1. 用户自行获取并粘贴政策原文，工具不负责内容版权与来源爬取。
2. 依赖可用的第三方大语言模型 API（或本地模型），其可用性、费用与内容策略由实现阶段评估。
3. 笔试演示环境需可访问模型服务或自带 mock（若允许）。
4. 依赖可用的**关系型数据库**及迁移/初始化方式（如 Alembic 或等价脚本）；`README` 中须写明如何建库与迁移。

## 10. 风险登记


| 风险         | 影响  | 缓解思路                              |
| ---------- | --- | --------------------------------- |
| 模型幻觉导致错误指引 | 高   | 结构化输出 + 强制「原文未提及」字段 + 免责声明 + 拒答规则 |
| 输入含敏感个人信息  | 中   | 隐私提示、按用户隔离存储、日志脱敏；可选不存原文全文仅存结构化结果 |
| 政策文本过长超上下文 | 中   | 长度限制、分段解读（后续迭代）                   |
| 凭据泄露、弱密码   | 高   | 密码哈希、HTTPS、最小密码长度提示、避免日志泄露令牌与密码 |
| 水平越权查看他人历史 | 高   | 历史接口强制鉴权且按 `user_id` 过滤；禁止仅靠前端传参指定 owner |

## 11. 需求范围小结

**本期包含**：用户注册与登录、会话鉴权、**关系型数据库**持久化用户与解读历史、文本粘贴输入、政策白话结构化解读、适老化展示、合规提示、FastAPI + Vue 技术实现路径。

**本期不包含**：政务数据对接、代替官方资格认定与待遇核算、全站政策爬虫与更新、官方认证背书、第三方 OAuth 登录（微信/支付宝等，可选后续迭代）。
